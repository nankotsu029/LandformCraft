# Task capsule V2-12-04 — v1→v2 migration tool

> 補助資料。正本は `docs/design-v2/task-index.md` §12 と `docs/adr/0035-v1-retirement-governance.md`。

## 1. Task（Index抜粋）

- 目的: 既存v1資産（intent／design package／Release 1）を明示操作でv2 artifactへ変換するmigration toolを実装する。前提 `V2-12-02`（完了）。
- Scope: v1 strict read→v2 contract mapping（欠落を推測補完しない）、新artifact生成（in-place書換なし）、変換reportとstrict verify、CLI subcommand、user docs。
- 非Scope: 自動一括migration、v1資産の削除・変更、lossy変換の暗黙容認、未知version受理。
- Gate: v1代表資産がv2で検証可能なartifactへ再現的に変換でき、非対応が明示報告され、D9とD2bのAcceptanceを満たせば完了。**変換にhard constraint推測が必要なら停止する。**

## 2. 設計判断（停止条件を回避した根拠）

v1 intentは `theme` 以外に幾何を持たない（zoneは `preferredArea` の粗い方位ヒントと `areaShare` のみ）。
v2 `Feature` はnormalized geometryを必須とするため、**v1からv2 Featureは導出できない**。
導出すればhard constraintの推測になり停止条件に該当する。したがって:

- 正確に写せるものだけを写す（`theme`、単一値enumのcoordinate system、provenance）。
- 写せないものは **1件ずつ report へ列挙**し、v2 intentには入れない（draft／未指定のまま）。
- lossyな変換は既定で **拒否**し、`accept-lossy` を明示指定した場合だけ実行する（非Scope「lossy変換の暗黙容認」の裏返し）。
- Release 1のblock artifact（tile／structures／assets／previews／world-blueprint v1）はv2 Blueprintの
  module／stage／field descriptorを持たないため変換不能。reportで「`lfc v2 export` による再生成が必要」と明示する。

`TerrainIntentV2.ProvenanceSource.UPGRADED_V1` と `DesignPathKindV2.IMPORT` は既存contractに存在し、
この経路のために用意されている。新しいenum値・contract変更は不要。

出力は **Release 2 design package**（`DesignArtifactVerifierV2` = Release 2 design package strict verifier）
＋ migration report のbundle。D9-10の「Release 2 strict verifierで検証」はこれで満たす。

## 3. D9品質条件 → 実装の対応

| # | 条件 | 実装 |
|---|---|---|
| 1 | 非破壊 | sourceへ一切書かない。testでsource treeのdigest不変を確認 |
| 2 | dry-run | `v2 migrate inspect`（`LegacyMigrationRequestV2.dryRun`） |
| 3 | 非overwrite | target bundle既存なら`IOException` |
| 4 | determinism | jobId=`nameUUIDFromBytes`(source digest)、timestampはv1 audit由来かEPOCH |
| 5 | report | `migration-report-v2.json`（新Schema） |
| 6 | version＋checksum | `sourceSchemaVersion`／`sourceDigest`（byte digest）／`sourceCanonicalChecksum`（Release 1 manifest checksum、無い資産は空文字） |
| 7 | 部分publish禁止 | staging→strict read-back→atomic move→published read-back |
| 8 | unknown version／未定義field拒否 | strict codec（schema `const` schemaVersion＋`additionalProperties:false`） |
| 9 | corrupted fail closed | v1 checksum／verifier経由でのみ読む |
| 10 | 結果のR2 strict verify | `DesignArtifactVerifierV2` ＋ bundle checksums read-back |

## 4. D2b Acceptance 6

「R4／R7削除後を模した状態でのpackaged-JAR migration test」:
`src/main/resources/legacy/v1/contracts/` へv1 contract schemaのimmutable copyを先置きし、
`StructuredDataValidator` を `/schemas/<name>` → `/legacy/v1/contracts/<name>` のfallback解決にする。
testは `/schemas/` のv1 entryとv1 orchestration classを隠すClassLoaderでmigrationを実行する。
active `schemas/` からの削除自体は `V2-12-06`（R7）の担当であり本Taskでは行わない。

## 6. 凍結対象

v1 Schema／generator `3.0.0-phase6`／Release format 1／v1 placement／v1 Undo／既存checksum／golden。
v1 command・v1 routingは触らない。catalogの能力・寸法昇格は行わない。

## 7. 停止条件

- v1→v2でhard constraintの推測が必要になる（→ 上記のreport化で回避）
- v1資産の変更・削除が必要になる
- Release 2 core capabilityやartifact formatの意味変更が必要になる
